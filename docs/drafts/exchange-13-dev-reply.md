## Surprises worth recording now (Continue / Load / Options)

- **Continue + Load Game are blocked on a client campaign-save layer that
  does not exist.** The engine has the serialize primitive
  (`engine/world-save.ts`, `WorldState` ↔ JSON) and the world-loop
  _harness_ persists saves to disk — but the **browser client has no
  persistence**: no localStorage wiring, no save list, no resume.
  `System.tsx` literally notes the format "awaits a UI." So:
  - **Continue** needs the client to snapshot `WorldState` to localStorage
    after each loop iteration and read it on load — a small new client
    save surface (engine primitive exists; client persistence + the
    sub-label metadata read do not).
  - **Load Game** needs a multi-slot store + the picker UI (your forward
    surface).
  - Note the two save concepts are distinct: the **campaign-save**
    (`world-save.ts`, a `WorldState` snapshot) is what Continue/Load mostly
    need; the **§7.13 mid-scenario save** (`scenario-save.ts`, input-stream
    replay) is the deeper resume-mid-fight primitive. Continue's "resume
    into the Hill" only needs the former.
- **Options/settings:** no client settings system exists at all (no audio,
  no settings store). The stub no-op is correct; scaffolding is trivial
  when wired. No surprise beyond "it's all forward."
